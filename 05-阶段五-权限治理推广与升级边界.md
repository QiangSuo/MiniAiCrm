# 阶段五：权限治理、推广与升级边界

## 1. 阶段目标

在轻量版闭环可用后，逐步推广到更多客户，并建立权限治理、数据质量巡检、运营机制和升级判断标准。

本阶段不是立刻升级重架构，而是回答三个问题：

```text
能不能安全推广？
数据质量能不能持续维护？
什么时候才需要从多维表格升级到 PostgreSQL？
```

## 2. 建议周期

真实客户试运行 1-2 周起，后续持续分批推广。

本阶段更多是业务验收和运营治理，不是纯开发。Codex 负责修复问题、补充脚本、优化卡片和生成巡检；你负责判断是否能推广、是否需要调整字段和权限。

## 3. 推广批次

建议按批次推广，不要一次性覆盖 200 个客户。

| 批次 | 范围 | 目标 |
| --- | --- | --- |
| 批次 1 | 3-5 个典型客户 | 打磨 Base 字段、卡片、提案和问答 |
| 批次 2 | 10-20 个重点客户 | 验证模板复制和客户负责人培训 |
| 批次 3 | 50 个活跃客户 | 验证周报、提醒、看板和权限维护 |
| 批次 4 | 约 200 个客户 | 常态化运营 |

在 Codex 快速开发模式下，推荐先设三个目标口径：

| 目标 | 周期 | 范围 |
| --- | --- | --- |
| 技术 Demo | 7-10 天 | 内部演示，不要求真实客户稳定使用 |
| 可用 MVP | 2-4 周 | 3 个核心闭环可用 |
| 真实客户试运行 | 4-6 周 | 3-5 个客户作战群稳定试用 |

每批结束后评估：

- 客户负责人使用率。
- 提案确认率。
- 资料归档量。
- 逾期行动闭环率。
- Base 字段是否足够。
- 权限是否可维护。
- Agent 回答是否可信。

## 4. 权限治理

### 角色设计

第一版建议角色：

```text
平台管理员
部门管理者
客户负责人
客户团队成员
解决方案团队
产品研发协同人员
合规审计人员
只读观察者
```

### 权限原则

- 客户负责人可编辑自己负责客户。
- 客户团队成员可查看和补充指定客户。
- 部门管理者可查看跨客户汇总。
- 产品研发只看产品需求和脱敏资料。
- 合规审计可查看敏感资料和审计记录。
- 严格保密客户单独配置权限或单独 Base。

### 权限落地方式

第一版可以组合使用：

```text
飞书 Base 高级权限
客户表中的客户团队字段
后端 PermissionGuard
敏感客户单独空间
```

不要只依赖提示词控制权限。

### 权限巡检

每月检查：

- 离职或转岗人员是否仍有权限。
- 客户团队是否变化。
- 严格保密客户权限是否过宽。
- 管理者跨客户权限是否合理。
- Agent 后端权限配置是否与 Base 权限一致。

## 5. 数据质量治理

### 数据质量指标

按周检查：

```text
客户是否有负责人
客户是否有作战群和 Wiki
资料证据是否有客户和来源
关键人是否有角色和关系负责人
商机是否有阶段和负责人
高金额商机是否有预算证据
行动承诺是否有负责人和截止日期
风险是否有责任人和应对动作
变更提案是否长期待确认
文档处理任务是否长期解析中
复杂文档是否解析失败
需人工处理资料是否无人跟进
```

### 数据质量视图

在 Base 中建立：

```text
无负责人客户
无作战群客户
无 Wiki 客户
无来源资料
无负责人行动
逾期行动
无证据商机
长期待确认提案
高风险无应对动作
长期解析中文档
解析失败文档
需人工处理资料
```

### 巡检流程

```text
每周定时扫描
        ↓
生成数据质量问题列表
        ↓
推送给客户负责人或平台管理员
        ↓
用户补充或修正
        ↓
下周复查
```

## 6. 运营机制

### 每周运营

- 检查待确认提案。
- 检查逾期行动。
- 检查高风险事项。
- 检查本周未更新的重点客户。
- 抽样检查 Agent 回答质量。

### 每月运营

- 复盘客户负责人使用情况。
- 优化 Base 字段和视图。
- 汇总常见提案修改原因。
- 评估是否需要新增资料类型、风险类型或商机阶段。
- 权限巡检。

### 每季度运营

- 评估是否扩展客户范围。
- 评估是否需要迁移 PostgreSQL。
- 评估是否需要引入专业文档检索。
- 评估是否与 CRM、合同、项目系统集成。

## 7. 升级到 PostgreSQL 的判断标准

不要一开始上 PostgreSQL，但需要提前知道什么时候该升级。

出现以下情况时，再考虑升级：

```text
Base 权限维护明显困难
跨客户统计越来越复杂
Agent 写入频率明显增高
多系统同步冲突频繁
数据量导致 Base 查询变慢
需要严格事务一致性
需要完整事件溯源和回滚
需要独立管理后台
需要复杂 BI 和多维分析
需要和 CRM/合同/项目系统做深度双向同步
```

升级后，飞书多维表格仍可保留为：

```text
业务录入界面
人工确认界面
轻量审核界面
管理者查看界面
```

PostgreSQL 则成为：

```text
后台事实源
权限和审计源
复杂分析源
系统集成源
```

## 8. 升级到专业检索系统的判断标准

第一版用资料证据表、摘要、关键词、Wiki 和云文档检索即可。

出现以下情况时，再考虑 RAGFlow、向量库或搜索引擎：

```text
文件数量快速增长
PDF/PPT/Word 格式复杂
用户经常需要跨大量资料深度问答
Agent 回答需要精确引用原文段落
资料摘要不足以支撑分析
需要按密级、客户、时间、资料类型复杂过滤
文档处理任务长期积压
解析失败或需人工处理比例过高
```

## 9. 升级到独立管理后台的判断标准

第一版用 Base 视图和飞书卡片，不做后台。

出现以下情况时，再考虑后台：

```text
Base 视图无法承载复杂操作
需要复杂权限配置界面
需要批量客户初始化
需要复杂数据质量修复工作台
需要管理者驾驶舱
需要审计报表和操作追踪
```

## 10. 阶段验收标准

- 已形成客户推广批次计划。
- 已有客户初始化模板。
- 已有客户负责人使用说明。
- 已有权限角色和巡检机制。
- 已有数据质量视图。
- 已有周/月/季度运营节奏。
- 已明确何时升级 PostgreSQL、专业检索和独立后台。
- 3-5 个真实客户完成至少 1-2 周试运行。
- 试运行期间的高频问题已整理为 Codex 修复任务。

## 11. Codex 开发任务拆分

| 任务 | 交付物 | 人工验收点 |
| --- | --- | --- |
| 试运行问题归类 | 问题台账、优先级、修复建议 | 是否覆盖真实使用问题 |
| 数据质量巡检脚本 | 无负责人、无证据、逾期等扫描 | 能发现真实脏数据 |
| 文档解析巡检脚本 | 长期解析中、解析失败、需人工处理资料 | 能发现复杂文档处理瓶颈 |
| 权限检查清单 | 角色、客户、密级、例外客户检查 | 是否符合最小可见原则 |
| 卡片体验优化 | 简化确认卡、修改卡、提醒卡 | 客户负责人是否愿意使用 |
| 周报格式优化 | 更短、更聚焦的周报模板 | 是否减少人工整理时间 |
| 推广材料生成 | 使用手册、初始化模板、FAQ | 新客户负责人能看懂 |

## 12. 交付物

- `推广批次计划`
- `客户初始化模板`
- `客户负责人使用手册`
- `权限角色说明`
- `数据质量巡检视图`
- `周运营检查清单`
- `月度复盘模板`
- `升级判断清单`

## 13. 阶段风险

| 风险 | 应对 |
| --- | --- |
| 推广过快导致数据乱 | 按批次推广，每批通过验收再扩大 |
| 权限维护跟不上 | 客户负责人、客户团队和敏感级别统一在客户表维护 |
| Base 变成杂乱表格 | 固定字段、视图、枚举和数据质量巡检 |
| Agent 回答质量波动 | 保留来源引用和反馈机制 |
| 迟迟不升级导致瓶颈 | 用升级判断清单定期评估 |
