# 阶段三：三个核心闭环 MVP

## 1. 阶段目标

在飞书多维表格数据底座和统一 Agent 后端可用后，优先实现三个最有价值的业务闭环：

```text
发材料 -> 先保全原始文件，写入资料证据表，再按复杂度异步解析
报进展 -> 生成变更提案，确认后写入 Base
问客户 -> 基于 Base 和证据回答
```

阶段完成后，3-5 个典型客户团队应能在真实飞书作战群中使用系统。

## 2. 建议周期

5-10 天。

本阶段不追求功能大而全。Codex 每次只实现一个闭环或一个卡片动作；你每天用真实或接近真实的客户样例验收，确认“业务能不能用”。

## 3. 闭环一：发材料

### 用户场景

用户在客户作战群上传文件，并说：

```text
这是今天客户给的集团数字化规划。
```

### 系统流程

```text
飞书群收到文件
        ↓
后端识别 chat_id -> customer_id
        ↓
获取文件链接和基础信息
        ↓
生成资料标题、资料类型、密级建议和文档复杂度
        ↓
写入变更提案表：资料归档提案
        ↓
创建文档处理任务，资料证据状态=待解析/解析中
        ↓
发送飞书资料归档确认卡
        ↓
用户确认
        ↓
写入资料证据表
        ↓
简单文档：生成摘要、关键片段、候选事实
复杂文档：异步解析，完成后再发送候选事实确认卡
        ↓
可选：更新客户 Wiki 的资料索引
```

### MVP 能力

- 支持文本说明 + 文件上传。
- 支持 PDF、Word、PPT、图片的基础文件信息记录。
- 第一版必须先保全原始文件和资料证据，不强求复杂文档即时解析完美。
- 原始文件保留在飞书云盘或原消息附件中。
- 资料证据表保存来源链接、提交人、提交时间、客户、密级、文档复杂度、解析状态、摘要。
- 文档处理任务表保存解析进度、错误原因和解析产物链接。

### 资料归档确认卡内容

```text
客户：中国XX集团
资料标题：集团数字化规划
资料类型：客户正式材料
密级建议：客户团队可见
文档复杂度：复杂 / 表格密集 / 未知
解析状态：待解析
摘要：……
来源：飞书作战群文件

[确认归档] [修改分类] [仅保存原始记录] [取消]
```

### 写入表

- 变更提案。
- 资料证据。
- 文档处理任务。
- 文档片段索引。

### 验收标准

- 用户上传资料后，能在资料证据表看到记录。
- 资料记录包含客户、来源、提交人、文件链接、资料类型、密级。
- 复杂文档有文档处理任务，且解析状态可追踪。
- 解析失败时原始资料仍然保留，并标记为需人工处理。
- 问客户时可以引用该资料。
- 取消或拒绝后不写入正式资料证据表。

## 4. 闭环二：报进展

### 用户场景

用户在客户作战群说：

```text
今天见了李总。他表示集团今年先做财务和采购，预算大约在一亿元左右，要求我们月底前提交整体方案。
```

### 系统流程

```text
收到进展消息
        ↓
识别客户上下文
        ↓
抽取接触记录、关键人、商机线索、行动承诺
        ↓
查询 Base 中已有关键人、商机、行动
        ↓
检测冲突
        ↓
生成变更提案
        ↓
飞书卡片展示拟写入内容
        ↓
用户确认 / 修改 / 拒绝 / 仅保存原始记录
        ↓
确认后写入接触记录、关键人、商机、行动承诺
```

### 需要抽取的信息

```text
客户
接触时间
客户参与人
接触摘要
涉及主题
预算线索
商机阶段变化
我方承诺
客户承诺
下一步行动
截止日期
来源证据
```

### 变更提案卡内容

```text
准备记录以下内容：

客户：中国XX集团
接触时间：2026-07-14
关键人：李总
涉及范围：财务、采购
预算线索：约 1 亿元
我方承诺：月底前提交整体方案

新增行动：
- 编制整体方案
- 截止日期：2026-07-31
- 负责人：待指定

待确认：
- 李总职位
- 行动负责人
- 预算是否已验证

[确认写入] [修改] [仅保存原始记录] [取消]
```

### 写入表

- 变更提案。
- 资料证据：保存原始消息作为证据。
- 接触记录。
- 关键人。
- 商机。
- 行动承诺。

### 冲突检测

至少检测：

- 已有同名关键人但职位不同。
- 商机预算已有不同金额。
- 商机阶段变更跨度过大。
- 同一行动已有不同截止日期。
- 资料密级建议与客户敏感级别冲突。

### 验收标准

- Agent 不直接写正式表，必须先生成提案。
- 关键人、预算、阶段、承诺、截止日期必须人工确认。
- 有冲突时必须展示冲突。
- 确认后能写入相关表。
- 每条正式记录能追溯到原始消息证据。

## 5. 闭环三：问客户

### 用户场景

用户在客户作战群问：

```text
这个客户目前最关键的三个突破口是什么？
```

或：

```text
我们还缺哪些关键人关系？
```

或：

```text
下周该推进什么？
```

### 系统流程

```text
收到问题
        ↓
识别客户上下文
        ↓
判断用户权限
        ↓
读取客户表、关键人表、商机表、接触记录、行动承诺、风险、资料证据
        ↓
组织回答上下文
        ↓
生成回答
        ↓
展示事实、单一来源信息、Agent 推断、建议
        ↓
附来源、数据截止时间、置信度、缺失信息
```

### 回答格式

回答必须包含：

```text
数据截止时间
结论摘要
支撑事实
单一来源信息
Agent 推断
建议动作
引用来源
缺失信息
```

### 示例回答结构

```text
数据截止：2026-07-14 18:00

最关键的三个突破口：

1. 财务共享方向
事实依据：7月14日接触记录提到客户今年先做财务。
证据：资料证据 E-001
置信度：中

2. 采购管理方向
事实依据：同一次接触记录提到采购也是优先范围。
证据：资料证据 E-001
置信度：中

3. 月底方案提交窗口
事实依据：我方承诺月底前提交整体方案。
对应行动：A-003，负责人待指定。

缺失信息：
- 李总具体职位未确认。
- 预算目前仍为单一来源信息。
- 财务部和采购部正式关键人尚未补齐。

建议动作：
- 明确方案负责人。
- 补充李总职位和决策影响力。
- 下次拜访覆盖财务部和采购管理部。
```

### 验收标准

- 回答不能跨客户引用资料。
- 回答不能把推断说成事实。
- 没有证据时必须说明“当前资料不足”。
- 回答必须能帮助客户负责人采取下一步行动。

## 6. 三个闭环的共用要求

### 客户隔离

每次执行必须带：

```text
customer_id
user_id
chat_id 或 conversation_id
operation
```

### 权限校验

读写前必须检查：

```text
用户是否属于客户团队
是否有资料密级权限
是否有跨客户分析权限
是否允许确认写入
```

### 证据追溯

正式记录必须能追溯到：

```text
原始消息
原始文件
提交人
提交时间
确认人
确认时间
```

### 错误处理

失败时必须：

- 不写入半成品正式记录。
- 更新变更提案状态为写入失败。
- 告知用户失败原因。
- 保留原始消息作为证据。

## 7. 阶段验收场景

| 场景 | 输入 | 预期 |
| --- | --- | --- |
| 发材料 | 上传客户 PDF | 生成资料归档提案，确认后写入资料证据表 |
| 复杂文档处理 | 上传扫描 PDF 或表格密集 PPT | 先保全原始文件，创建解析任务，完成后再出候选事实 |
| 报进展 | 一段拜访文字 | 生成变更提案，确认后写入接触、商机、行动 |
| 问客户 | 询问突破口 | 基于 Base 和证据回答，附来源和缺失信息 |
| 冲突处理 | 新预算与旧预算不同 | 卡片提示冲突，不静默覆盖 |
| 无权限访问 | 用户问未授权客户 | 返回权限不足，不泄露资料 |

## 8. Codex 开发任务拆分

| 顺序 | 任务 | 验收方式 |
| --- | --- | --- |
| 1 | 资料归档提案 | 上传资料后生成提案卡 |
| 2 | 资料确认写入 | 点击确认后写入资料证据表 |
| 3 | 文档处理任务 | 复杂文档有解析状态和任务记录 |
| 4 | 解析完成候选提案 | 简单文档生成摘要、片段和候选事实 |
| 5 | 报进展结构化抽取 | 输入拜访文本后生成结构化提案 |
| 6 | 报进展确认写入 | 写入接触记录、行动承诺、商机线索 |
| 7 | 冲突检测 | 预算或阶段冲突时卡片提示 |
| 8 | 客户问答只读工具 | 能读取客户、商机、行动、风险 |
| 9 | 基于证据回答 | 回答附来源、截止时间、缺失信息 |
| 10 | 权限回归 | 未授权用户不能跨客户查询或写入 |

## 9. 阶段风险

| 风险 | 应对 |
| --- | --- |
| 提案抽取不准 | 卡片确认、待确认字段、原文回链 |
| 用户觉得确认麻烦 | 只对关键字段强确认，普通摘要可一键确认 |
| 回答像泛泛建议 | 强制引用 Base 记录和证据 |
| 文件解析不稳定 | 第一版以归档和摘要为主，复杂解析后置 |
| Base 写入失败 | 先写提案，再写正式表；失败时提案保留 |
