# 轻量版总体路线图

## 1. 建设目标

建设一个轻量、可快速落地的 XERP 客户情报平台，用于管理约 200 个客户的客户资料、关键人、商机、接触记录、行动承诺、风险、证据和客户经营分析。

本版按“Codex 快速开发 + 人做业务验收”的模式推进：Codex 主要负责工程实现、文档、测试脚本和调试；人负责业务口径、飞书授权、真实数据确认和最终验收。

本版不追求一开始做成重型平台，而是优先在飞书生态内完成业务闭环：

```text
飞书客户作战群 / 机器人单聊
        ↓
飞书企业自建应用机器人
        ↓
统一 Codex Agent 后端
        ↓
飞书多维表格 Base：结构化事实库
飞书 Wiki / 云文档：客户展示页
飞书云盘 / 附件：原始资料库
```

## 2. 与重架构版本的差异

| 项目 | 重架构版本 | 本轻量版 |
| --- | --- | --- |
| 结构化事实库 | PostgreSQL | 飞书多维表格 |
| Agent 编排 | LangGraph 等专业框架 | 统一 Codex Agent 后端内置编排 |
| 文档检索 | RAGFlow / 向量库 | 第一版用资料证据表、摘要、关键词、Wiki/云文档检索 |
| 原始文件 | MinIO / 企业对象存储 | 飞书云盘 / Wiki 附件 |
| 长任务 | Temporal | 后端定时任务 / 轻量任务队列 |
| 管理后台 | 独立后台 | Base 视图、仪表盘、飞书卡片 |
| 目标 | 生产级平台 | 快速验证和轻量运营 |

## 4. 数据分层

| 层 | 工具 | 作用 |
| --- | --- | --- |
| 原始资料层 | 飞书云盘、Wiki 附件、云文档 | 保存 PDF、PPT、Word、截图、招投标材料、会议纪要 |
| 结构化事实层 | 飞书多维表格 | 保存客户、证据、关键人、商机、行动、风险、提案、解析任务状态 |
| 展示协作层 | 飞书 Wiki | 展示客户首页、客户背景、经营计划、分析报告 |
| Agent 编排层 | 统一 Codex Agent 后端 | 理解输入、生成提案、调用工具、组织回答 |
| 交互确认层 | 飞书机器人和卡片 | 收消息、发确认卡片、推送周报和提醒 |
| 文档处理层 | 统一 Codex Agent 后端内的解析流水线 | 复杂文档异步解析、摘要、分段、候选事实抽取 |

## 5. 分阶段路线

| 阶段 | 名称 | 建议周期 | 目标 |
| --- | --- | --- | --- |
| 阶段零 | Codex 快速开发准备 | 1-2 天 | 建项目骨架、开发约束、配置清单、验收样例和每日任务拆分 |
| 阶段一 | 飞书多维表格数据底座 | 2-4 天 | 建立 XERP 客户情报 Base、8 张核心表、字段、视图和基础权限 |
| 阶段二 | 统一 Codex Agent 后端与飞书接入 | 4-7 天 | 接入飞书机器人，建立事件网关、客户路由、Base 工具和卡片回调 |
| 阶段三 | 三个核心闭环 MVP | 5-10 天 | 跑通发材料、报进展、问客户 |
| 阶段四 | 周报提醒与经营看板 | 4-7 天 | 增加周报、逾期提醒、风险提醒、Base 视图和管理看板 |
| 阶段五 | 真实客户试运行、权限治理与推广 | 1-2 周起 | 3-5 个真实客户试运行，建立权限、数据质量和升级判断机制 |
| 横向专题 | Base 扩展性与复杂文档处理 | 与阶段一到三并行 | 预留解析任务、片段引用、迁移边界和未来检索升级接口 |

## 6. 周期口径

| 目标口径 | 建议周期 | 完成标准 |
| --- | --- | --- |
| 技术 Demo | 7-10 天 | 能演示 Base 读写、飞书收消息、提案卡片和简单问答 |
| 可用 MVP | 2-4 周 | 三个核心闭环可用，确认后能写 Base，问答可引用证据 |
| 真实客户试运行 | 4-6 周 | 3-5 个客户真实作战群稳定试用，周报和提醒能产生业务价值 |
| 扩大推广 | 6-10 周起 | 推广到 10-20 个重点客户，再逐步扩展到更多客户 |

## 7. 项目角色

| 角色 | 职责 |
| --- | --- |
| 你 / 业务负责人 | 定义客户经营目标、验证客户、业务口径、飞书授权、阶段验收和推广节奏 |
| Codex | 负责代码实现、文档生成、测试脚本、调试、重构、错误修复和阶段交付整理 |
| 客户负责人代表 | 提供真实资料、验证卡片内容、反馈使用体验 |
| 飞书管理员 | 协助自建应用、机器人入群、事件订阅、Base/Wiki 权限和应用授权 |
| 数据治理参与者 | 协助确认密级、权限、确认状态和数据质量巡检规则 |

最小协作模式是：

```text
你 + Codex
```

但若你没有飞书管理员权限，需要飞书管理员配合完成应用和权限配置。

## 8. 总体里程碑

### M0：Codex 开发环境可持续

- 项目骨架可启动。
- 飞书、Base、模型和运行环境配置项明确。
- 开发约束、验收样例和任务清单完成。

### M1：Base 数据底座可用

- 建成 `XERP客户情报库`。
- 完成 8 张核心表。
- 完成客户视图、待确认视图、逾期行动视图、高风险视图。
- 3-5 个典型客户有初始数据。

### M2：飞书机器人和 Agent 后端可用

- 机器人可接收群消息、单聊和文件。
- 后端可识别 `chat_id -> customer_id`。
- 后端可读写 Base。
- 飞书卡片确认链路可用。

### M3：三个核心闭环跑通

- 发材料可归档到资料证据表。
- 报进展可生成变更提案，确认后写入结构化表。
- 问客户可基于 Base 和证据回答。

### M4：经营自动化可用

- 自动客户周报可生成。
- 逾期行动和风险提醒可推送。
- 管理者能看重点客户、重点商机、风险和行动看板。

### M5：轻量运营稳定

- 推广到更多客户。
- 权限和密级规则可执行。
- 数据质量巡检常态化。
- 明确是否需要升级 PostgreSQL 或专业检索系统。

## 9. 本版不做事项

第一版明确不做：

- PostgreSQL。
- 独立管理后台。
- 复杂微服务拆分。
- Kafka / Temporal。
- RAGFlow / 自建向量库。
- 200 个独立 Agent。
- Agent 直接静默改 Wiki 或 Base 正式表。
- 把大段全文、复杂文档全部切片和向量索引都塞进 Base。
- 复杂文档一次解析失败就丢弃或直接让 Agent 猜结论。

## 10. 成功标准

| 指标 | 成功口径 |
| --- | --- |
| 使用意愿 | 客户负责人愿意在真实作战群中发材料、报进展、问客户 |
| 客户隔离 | 群聊、单聊、查询、写入均不串客户 |
| 提案质量 | 业务人员能看懂并愿意确认变更提案 |
| 数据可追溯 | 关键事实能追溯到证据来源 |
| 行动闭环 | 逾期行动能被发现和提醒 |
| 周报价值 | 周报能减少人工整理客户进展的时间 |
| 轻量可维护 | 不依赖复杂后台，业务和技术团队都能维护 |
| Codex 可持续开发 | 每个开发任务都有输入、验收样例和可运行测试，不依赖一次性大改 |
