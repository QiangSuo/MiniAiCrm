# Base 扩展性与复杂文档处理方案

## 1. 为什么要单独设计

轻量版使用飞书多维表格作为结构化事实库，是为了快速落地。但客户资料系统天然会增长：

- 客户数量会从 3-5 个验证客户扩展到约 200 个客户。
- 字段会从最小经营字段扩展到 CRM、合同、项目、产品需求字段。
- 文档会从简单会议纪要扩展到复杂 PDF、PPT、Excel、扫描件、招投标文件和大篇幅方案。
- 问答会从摘要级回答扩展到精确引用、跨文档比较和复杂分析。

因此第一版虽然要轻，但不能把路走死。

本方案的核心原则是：

```text
Base 做轻量事实库和处理状态台账，
不要把 Base 做成全文仓库、向量库或复杂文档解析引擎。
```

## 2. Base 的可扩展性原则

### 原则一：业务 ID 和飞书 record_id 分离

每张核心表都要有稳定业务 ID：

```text
客户ID
证据ID
接触ID
关键人ID
商机ID
事项ID
风险ID
提案ID
任务ID
片段ID
```

飞书 `record_id` 只作为飞书内部记录定位，不作为长期业务主键。未来迁移 PostgreSQL 时，业务 ID 可以继续沿用。

### 原则二：字段映射配置化

后端不能在代码里硬编码飞书字段 ID。

建议维护配置：

```yaml
tables:
  customers:
    table_id: tbl_xxx
    fields:
      customer_id: fld_xxx
      customer_name: fld_xxx
      owner: fld_xxx
  evidence:
    table_id: tbl_xxx
    fields:
      evidence_id: fld_xxx
      parse_status: fld_xxx
      parse_job_id: fld_xxx
```

这样未来字段重命名、复制 Base、迁移环境时，主要改配置，不改业务代码。

### 原则三：核心表稳定，扩展表承接增长

第一版核心表保持 8 张：

```text
客户
资料证据
接触记录
关键人
商机
行动承诺
风险
变更提案
```

为了未来扩展，预留 4 张扩展表：

```text
文档处理任务
文档片段索引
字典配置
操作日志
```

复杂文档、解析状态、字段枚举和审计不要全部塞进核心表。

### 原则四：Base 存摘要和引用，不存全文

复杂文档可能有几百页，或者包含大量表格、图片、扫描页。Base 不适合存完整全文切片。

Base 只存：

```text
原始文件链接
资料元数据
解析状态
摘要
关键引用片段
解析产物链接
片段索引
候选事实
确认状态
```

全文、解析 JSON、OCR 结果、大段中间产物应存到：

```text
飞书云盘
飞书云文档
后续对象存储
后续专业检索系统
```

### 原则五：所有抽取结果先成为候选

复杂文档里抽取出的客户事实、关键人、商机、风险和行动，不能直接写正式表。

必须先进入：

```text
变更提案
```

经用户确认后，再写入：

```text
接触记录
关键人
商机
行动承诺
风险
```

## 3. 推荐 Base 扩展结构

### 必须增强：资料证据表

在原字段基础上增加：

```text
文档复杂度
解析状态
解析任务ID
解析产物链接
解析版本
解析错误
```

文档复杂度：

```text
简单
复杂
扫描件
表格密集
图片密集
受保护
未知
```

解析状态：

```text
未解析
待解析
解析中
部分完成
解析失败
已解析
需人工处理
```

### 扩展表一：文档处理任务

用于跟踪复杂文档异步处理过程。

关键字段：

```text
任务ID
客户
资料证据
文件类型
文档复杂度
处理状态
当前步骤
解析产物链接
片段数量
候选事实数量
错误信息
重试次数
解析版本
创建时间
完成时间
```

### 扩展表二：文档片段索引

用于保存关键页、章节、表格、片段和引用，不保存全文。

关键字段：

```text
片段ID
客户
资料证据
任务ID
页码/位置
片段类型
片段摘要
关键引用
关联实体
可信度
是否生成提案
```

### 扩展表三：字典配置

用于统一管理枚举值和业务口径。

例如：

```text
资料类型
密级
商机阶段
风险类型
行动状态
确认状态
文档复杂度
解析状态
```

### 扩展表四：操作日志

用于记录关键写入、解析失败、卡片确认和异常。

详细技术日志仍放后端日志系统，Base 只保存业务可见的关键操作。

## 4. 复杂文档处理总流程

```text
用户上传文档
        ↓
保存原始文件和来源信息
        ↓
写入资料证据表，状态=待解析
        ↓
创建文档处理任务
        ↓
文档复杂度判断
        ↓
异步解析
        ↓
生成摘要、关键片段、候选事实
        ↓
写入文档片段索引
        ↓
生成变更提案
        ↓
用户确认
        ↓
写入正式结构化表
```

关键点：

- 用户上传后立即返回“已收到，正在处理”，不要让用户等完整解析。
- 解析失败也保留原始资料。
- 解析失败不等于业务失败，可以进入人工处理。
- 候选事实必须通过确认卡片进入正式表。

## 5. 文档复杂度分类与处理策略

| 类型 | 例子 | 处理策略 |
| --- | --- | --- |
| 简单文档 | 普通 Word、短 PDF、文字型会议纪要 | 直接抽文本、摘要、候选事实 |
| 扫描件 | 扫描 PDF、图片版合同、拍照材料 | OCR 后再抽取，置信度降低 |
| 表格密集 | Excel、多页报价表、清单、招标评分表 | 先保留文件和 sheet 摘要，关键表格人工确认 |
| 图片密集 | PPT、架构图、组织架构截图 | 提取标题和图片说明，复杂图先人工复核 |
| 受保护文档 | 加密 PDF、权限受限文件 | 标记需人工处理，不做猜测 |
| 超长文档 | 几百页招投标文件、集团规划 | 分章节摘要，先抽目录、关键章节和高价值片段 |
| 格式混杂 | 压缩包、多文件组合材料 | 拆分成多个资料证据或一个资料包任务 |

## 6. 第一版如何实现，不做太重

第一版不需要一次上专业 RAG 或完整 OCR 平台。

可按三层实现：

### L1：保全和台账

必须做：

```text
保存原始文件
记录资料证据
标记文档复杂度
标记解析状态
记录解析失败原因
```

### L2：轻量解析

优先做：

```text
文件名和用户说明解析
简单文本提取
摘要生成
关键词提取
少量关键片段保存
候选事实提案
```

### L3：后续增强

等真实复杂文档多起来，再加：

```text
OCR
版面分析
表格抽取
PPT 图片理解
长文档分章节摘要
向量检索
RAGFlow 或专业文档解析服务
```

## 7. 后端需要预留的接口

统一 Codex Agent 后端里建议预留三个抽象接口：

```text
DocumentStorage
DocumentParser
RetrievalProvider
```

### DocumentStorage

负责：

```text
保存原始文件
获取文件链接
保存解析产物
返回解析产物链接
```

第一版实现：

```text
飞书云盘 / 飞书附件
```

未来可替换：

```text
MinIO / 企业对象存储
```

### DocumentParser

负责：

```text
识别文件类型
判断复杂度
抽文本
做摘要
抽候选事实
生成关键片段
```

第一版实现：

```text
简单解析 + 模型摘要 + 人工确认
```

未来可替换：

```text
OCR / 版面分析 / RAGFlow / 专业解析服务
```

### RetrievalProvider

负责：

```text
按 customer_id 检索资料
按密级过滤
返回引用片段
返回来源链接
```

第一版实现：

```text
资料证据表 + 文档片段索引表 + Wiki/云文档链接
```

未来可替换：

```text
向量库 / RAGFlow / OpenSearch / pgvector
```

## 8. 复杂文档场景下的用户体验

用户上传复杂文档后，机器人不应该假装马上读完。

推荐回复：

```text
已收到资料：《XX集团数字化规划.pdf》

我会先完成：
1. 保存原始文件
2. 建立资料证据记录
3. 判断文档复杂度
4. 生成初步摘要

这份资料可能需要较长时间解析。解析完成后，我会发送待确认事实卡片。
```

解析完成后：

```text
资料解析完成：

客户：中国XX集团
资料类型：客户正式材料
复杂度：表格密集 / 长文档
摘要：……
识别候选事实：8 条
需人工确认：5 条
解析不确定内容：2 条

[查看候选事实] [确认归档] [标记需人工处理]
```

解析失败时：

```text
资料已保存，但自动解析失败。

可能原因：
- 文件受保护
- 扫描质量较低
- 格式复杂

我已保留原始文件，并将状态标记为“需人工处理”。
```

## 9. 什么时候升级专业文档检索

满足以下任意条件时，建议升级：

```text
复杂 PDF/PPT/Excel 占比超过 30%
用户经常要求精确引用原文页码
资料数量超过 Base 摘要和人工视图可管理范围
问答经常因为资料过长而漏掉关键信息
需要跨多个长文档做比较分析
需要按段落、表格、图片进行检索
```

升级方向：

```text
RAGFlow
OpenSearch
pgvector
专业 OCR / 文档解析服务
企业对象存储
```

## 10. 什么时候升级 PostgreSQL

Base 可以继续作为业务界面，但出现以下情况时，后台事实源应考虑迁移 PostgreSQL：

```text
字段和关联关系变得复杂
多系统同步频繁
权限和审计要求提高
跨客户统计和查询性能不足
需要完整事件溯源和回滚
Agent 写入频率高
Base 视图和权限维护成本过高
```

迁移时，Base 保留为：

```text
业务录入和查看界面
人工确认界面
轻量审核界面
管理者协作界面
```

PostgreSQL 承担：

```text
后台事实源
复杂权限和审计
事件日志
跨系统同步
复杂分析
```

## 11. 对当前实施计划的影响

阶段一必须增加：

- 资料证据表扩展字段。
- 文档处理任务表字段设计。
- 文档片段索引表字段设计。
- 字典配置表字段设计。

阶段二必须增加：

- `DocumentStorage` 抽象。
- `DocumentParser` 抽象。
- `RetrievalProvider` 抽象。
- 异步解析任务状态更新。

阶段三必须调整：

- 发材料闭环先保证原始文件保全和资料证据记录。
- 复杂文档不要求即时完成解析。
- 解析完成后再生成候选事实提案。

阶段五必须增加：

- 文档解析失败率巡检。
- 需人工处理资料视图。
- 专业检索升级判断。
