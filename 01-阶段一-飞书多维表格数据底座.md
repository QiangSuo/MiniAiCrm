# 阶段一：飞书多维表格数据底座

## 1. 阶段目标

先搭建一个轻量、可被业务人员理解、可被 Agent 稳定读写的飞书多维表格 Base。

阶段完成后，应具备：

- 一个统一的 `XERP客户情报库`。
- 8 张核心表。
- 标准字段、枚举值、关联关系和视图。
- 3-5 个典型客户的初始数据。
- 可供 Agent 后端调用的基础数据结构。

## 2. 建议周期

2-4 天。

本阶段由 Codex 负责输出字段字典、建表清单、样例数据和后端字段映射配置；你负责确认字段是否符合客户经营习惯，并完成飞书授权或协助 Codex 调用飞书接口建表。

## 3. Base 总体设计

Base 名称：

```text
XERP客户情报库
```

核心表：

```text
1. 客户
2. 资料证据
3. 接触记录
4. 关键人
5. 商机
6. 行动承诺
7. 风险
8. 变更提案
```

扩展预留表：

```text
9. 文档处理任务
10. 文档片段索引
11. 字典配置
12. 操作日志
```

第一版必须建 8 张核心表。扩展预留表可以先只设计字段，不一定立刻启用；但字段命名和后端接口要提前预留，避免未来复杂文档和迁移时推倒重来。

设计原则：

- 一个统一 Base 管理大部分客户。
- 每条业务记录必须关联一个客户。
- 敏感客户可单独 Base 或单独权限空间处理。
- 字段先少后多，第一版只保留高频经营字段。
- 公式、lookup、仪表盘可以后置，不在第一天追求完整。
- 所有表必须保留稳定业务 ID，不依赖飞书 record_id 作为唯一业务 ID。
- 后端使用逻辑字段名到飞书 field_id 的映射配置，不在代码里硬编码字段 ID。
- Base 存事实、状态、摘要和引用，不存复杂文档全文。
- 原始文件、解析产物和大段文本放飞书云盘、云文档或后续对象存储，Base 只存链接和处理状态。

## 4. 通用字段标准

所有业务表建议尽量保留以下通用字段：

```text
客户
创建人
创建时间
最后更新人
最后更新时间
来源证据
确认状态
密级
备注
```

所有核心表建议增加的可扩展字段：

```text
外部系统ID
数据版本
数据来源系统
最后同步时间
是否已迁移
迁移批次
```

这些字段第一版可以为空，但保留字段位置有助于未来接入 CRM、迁移 PostgreSQL 或做数据校验。

确认状态枚举：

```text
待确认
已确认
已拒绝
有冲突
已归档
已作废
```

密级枚举：

```text
内部公开
客户团队可见
部门管理层可见
严格保密
```

资料类型枚举：

```text
客户正式材料
会议纪要
拜访记录
招投标材料
政策公告
组织架构
方案材料
邮件截图
微信截图
竞争情报
产品需求
其他
```

## 5. 表结构设计

### 表 1：客户

用途：

- 管理客户主数据。
- 支撑客户作战群、Wiki、权限和跨客户看板。

建议字段：

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| 客户ID | 文本 | 唯一标识，例如 `customer_0001` |
| 客户名称 | 文本 | 客户正式名称 |
| 客户简称 | 文本 | 群名和日常称呼 |
| 客户类型 | 单选 | 央企集团、央企二级公司、地方国企、民企、谨慎保留、例外客户 |
| 行业 | 单选/多选 | 能源、装备、交通、金融等 |
| 区域 | 单选 | 华北、华东、华南等 |
| 集团属性 | 单选 | 总部、二级公司、子公司、独立法人 |
| 客户负责人 | 人员 | 第一责任人 |
| 客户团队 | 人员多选 | 可访问和协作人员 |
| 敏感级别 | 单选 | 内部公开、客户团队可见、部门管理层可见、严格保密 |
| 客户Wiki链接 | 超链接 | 客户首页 |
| 客户作战群ID | 文本 | `chat_id -> customer_id` 映射依据 |
| 当前经营状态 | 单选 | 活跃、培育、推进中、暂停、成交、丢单 |
| 最后更新时间 | 日期 | 可由系统维护 |

建议视图：

- 全部客户。
- 我的客户。
- 重点客户。
- 谨慎保留客户。
- 无 Wiki 客户。
- 无作战群客户。

## 6. 初始客户验证

选择 3-5 个客户：

- 一个大型央企集团。
- 一个央企二级子公司。
- 一个地方国企。
- 一个谨慎保留或敏感客户。
- 可选一个正在推进中的高价值商机客户。

每个客户至少录入：

- 客户主数据 1 条。
- 关键人 3-5 条。
- 商机 1-3 条。
- 接触记录 3-5 条。
- 行动承诺 3-5 条。
- 风险 1-3 条。
- 资料证据 5-10 条。

## 7. 阶段验收标准

- `XERP客户情报库` 已创建。
- 8 张核心表已创建。
- 关键枚举值已统一。
- 3-5 个客户已有初始数据。
- 可以按客户筛选资料、商机、行动、风险。
- 可以看到待确认提案、逾期行动、高风险客户视图。
- Agent 后端后续可通过表名、字段名、视图和记录 ID 读写数据。
- Base 表 ID、字段 ID 已整理成后端配置文件或配置清单。
- 至少有 3 个验收样例客户可用于后续端到端测试。
- 资料证据表已包含解析状态、文档复杂度、解析任务ID、解析产物链接等扩展字段。
- 文档处理任务表和文档片段索引表至少完成字段设计，复杂文档场景不需要改核心表即可接入。

## 8. Codex 开发任务拆分

| 任务 | 交付物 | 人工验收点 |
| --- | --- | --- |
| 输出 Base 字段字典 | 8 张表字段清单、枚举值、关联关系 | 字段是否足够支撑客户经营 |
| 输出建表执行清单 | 建表顺序、字段类型、视图清单 | 是否能按清单在飞书中创建 |
| 生成测试数据 | 3-5 个客户样例数据 | 样例是否接近真实业务 |
| 整理字段映射配置 | table_id / field_id 配置模板 | 后端能否稳定读写 |
| 建立数据质量视图 | 无负责人、无证据、逾期等视图 | 视图是否能发现真实问题 |

## 9. 阶段风险

| 风险 | 应对 |
| --- | --- |
| 字段过多，业务人员不愿填 | 第一版只保留高频字段，复杂字段放备注或后续补充 |
| 表结构经常改，Agent 写入不稳定 | 冻结 MVP 字段，字段调整走变更记录 |
| 权限一开始设计过复杂 | 先按验证客户和角色配置基础权限，敏感客户单独处理 |
| 多维表格被当成普通 Excel | 明确字段类型、枚举值、确认状态和证据来源 |
