# XERP 客户情报平台轻量版实施计划

本目录是基于“统一 Codex Agent 后端 + 飞书多维表格 + 飞书 Wiki”的新版轻量路线拆解，并已按“Codex 快速开发 + 人做业务验收”的方式重新调整周期和推进节奏。

与上一版重架构不同，本版第一阶段不建设 PostgreSQL、RAGFlow、MinIO、Kafka、Temporal 或独立管理后台，而是优先利用飞书生态完成业务闭环。


> [!IMPORTANT]
> **2026-07-15 Demo 紧急路径已启用。** 当前仓库尚无应用代码，今晚不再按原 7—10 天阶段顺序实施，而是先由 Codex CLI 自主完成一个本地、无密钥也可运行的纵向切片。唯一执行入口见 [07-2026-07-15-CodexCLI自主开发Demo计划.md](./07-2026-07-15-CodexCLI自主开发Demo计划.md)。原 `00`—`06` 文档继续作为 Demo 后的产品路线和领域口径。

## 新版路线一句话

```text
飞书做入口和协作界面，
多维表格做轻量结构化事实库，
Wiki 做客户展示页，
统一 Codex Agent 后端做理解、提案、确认、工具调用和自动化。
```

## 本版核心原则

- 第一版不建设 PostgreSQL，多维表格就是轻量事实库。
- 不把 Wiki 当唯一数据库，Wiki 只做人读的展示层。
- 不做 200 个 Agent，只做一个统一 Agent 后端，靠 `customer_id` 隔离。
- 不让 Agent 直接静默改客户资料，所有写入先形成变更提案。
- 不把复杂业务逻辑塞进 Base workflow，核心逻辑放在统一 Agent 后端。
- 不追求一次性全自动，先把业务人员愿意用的闭环跑通。
- Base 只存结构化事实、证据元数据、处理状态和关键引用，不存大段全文。
- 复杂文档必须先保全原始文件，再异步解析，再生成候选提案，不能假设一次解析成功。

## Codex 快速开发节奏

本计划不再按传统小团队排期估算，而按“Codex 负责快速工程实现，你负责业务验收、飞书授权、真实数据确认”的模式推进。

推荐目标分成两条轨道：

```text
2026-07-15 本地纵向 Demo：今晚开发、明日彩排和演示
正常技术 Demo：7-10 天
可用 MVP：2-4 周
真实客户试运行：4-6 周
```

明日 Demo 只冻结五个动作：发材料、报进展、提案确认、问客户、经营概览。真实飞书接入、复杂文档解析和生产数据治理不得阻塞这条路径。

推荐推进顺序：

1. 先做阶段零，让 Codex 有稳定项目骨架、开发约束和配置清单。
2. 用 2-4 天搭好 Base 数据底座。
3. 用 4-7 天接好飞书机器人、卡片回调和统一 Agent 后端。
4. 用 5-10 天跑通“发材料、报进展、问客户”三个核心闭环。
5. 用 4-7 天补齐周报、提醒和轻量看板。
6. 并行补齐 Base 扩展字段、文档处理任务表和复杂文档处理规则。
7. 用 1-2 周做 3-5 个真实客户试运行和业务验收。
8. 当 Base 成为瓶颈时，再迁移到 PostgreSQL 或增加专业检索组件。
